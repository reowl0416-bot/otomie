# SPEC.md — 楽譜スクショ 音名・コード判定アプリ 仕様書

本書はヒアリングで確定した前提・要件・判断基準をまとめた確定仕様である。
実装計画は [TASKS.md](TASKS.md)、開発ガイドは [CLAUDE.md](CLAUDE.md) を参照。

---

## 1. 目的とユースケース

### 1.1 目的
楽譜のスクリーンショットを読み込み、各音符の音名（C4, D4 …）と、和音から推定したコード名を、元画像に重ねて表示する。

### 1.2 主ユースケース
- **対面 / 画面共有でのライブ指導**。教える側がスクショを一枚貼り、その場で音名・コードを見せながら説明する。
- ワークフローは「**一枚貼り → 注釈表示**」。リアルタイム映像解析は行わない。

### 1.3 想定ユーザー
- 音楽を人に教える立場の制作者。楽譜を読むのが苦手な相手にも「どの音符が何の音か」を目で追わせたい。

---

## 2. スコープ

### 2.1 対象
- 入力：**静止画一枚**（PNG / JPG）のアップロード / 貼り付け。
- 譜面：**大譜表（トレブル＋バス）のピアノ譜**。きれいにレンダリングされた五線譜（例：YouTube「楽譜ダウンロード」系動画のスクショ。坂本龍一「Aqua」等）。
- 譜面フォーマットは**ほぼ一定**。特定フォーマットに特化したチューニングを許容する。

### 2.2 非対象（今回のスコープ外）
- 紙のスマホ撮影（傾き・影あり）、手書き譜。
- 動画URLからの取り込み・フレーム抽出（YouTube規約・著作権・処理コストの懸念のため。将来検討）。
- 音の再生。
- 注釈付き譜の書き出し / エクスポート（ライブ用途のため。将来検討）。
- ドレミ併記・C3方式オクターブ（将来のオプション）。

---

## 3. 入力仕様

- 受け付ける形式：PNG / JPG の画像ファイル。ファイル選択・ドラッグ&ドロップ・クリップボード貼り付けに対応。
- 画像は**外部に送信しない**。ブラウザ内でのみ処理する。
- 画像はズーム / パンして拡大確認できる。

## 4. 音名判定仕様

### 4.1 表記
- 英米式ピッチクラス（C, D, E, F, G, A, B）＋ オクターブ番号。例：`C4`, `F#4`, `B♭3`。
- **オクターブ定義：真ん中のC = C4**（科学的音高表記, MIDI ノート 60）。この定義は将来 C3 方式に切替可能な設計余地を残すが、初期はC4固定。

### 4.2 音高の決定
音名は「五線上の垂直位置 × 音部記号 × 調号 × 臨時記号」で一意に決まる。

- **音部記号**：トレブル / バスを段ごとに把握（大譜表なので上段トレブル・下段バスが基本だが、決め打ちにせず把握する）。
- **調号**：譜頭の♯/♭の数から調を把握し、各音名に既定の変化を反映。
- **臨時記号**：小節内で書かれた♯/♭/♮を反映（同小節・同オクターブに継続、小節線でリセットの原則）。
- **加線（ledger line）**：五線外の音も高さを算出する。

### 4.3 異名同音（黒鍵）の表記規則
- **調号・臨時記号に従う**。譜面のキーや書かれた変化記号に一致させる（例：D長調では G♭ ではなく F#）。
- 常時シャープ統一や両方併記はしない（初期方針）。
- ∴ 正しい表記には調号・音部記号の把握が前提。把握できない/誤る場合は、修正UI（第7章）で人が訂正する。

## 5. コード判定仕様

### 5.1 音のまとめ方（グルーピング）
- **拍 / 小節を単位に、その区間に鳴る音を集めて和声を推定する**。
- 理由：対象曲（Aqua 等）はアルペジオ主体で、真に縦に重なる音が少ないため、縦の同時音だけではコードがほとんど出ない。
- 区間の取り方には曖昧さが残ることを許容し、必要に応じて後述の候補提示・修正で補う。

### 5.2 コード名の流儀
- **ポップス表記**（例：`C`, `Am7`, `G7`）。メジャー=大文字のみ、マイナー=`m`。
- **分数 / オンベース表記**：最低音がルートでない場合は `C/E` のようにベース音を併記する（大譜表の左手＝ベースを活かす）。
- **テンションも詳細に**：9th / sus4 / add9 などの拡張コードも、判定可能な範囲で表記する。
- **候補提示**：区間内の音から複数の解釈が成り立つ場合は、単一に決め打ちせず候補を提示する。

### 5.3 非和声音の扱い
- 経過音・刺繍音などが混じる前提で、区間内の主要な構成音から最も妥当な和声を推定する。確信が持てない区間は候補提示 or 空欄（＋手動指定）とする。

## 6. 認識（画像処理 / CV）仕様

### 6.1 方針
- **ブラウザ内 CV**（無料・ローカル・送信なし）。半自動：自動認識で下地を作り、**人による確認・修正を必須**とする。
- クラウド画像認識/OMR API は使わない（事前承認なしの課金を避ける方針）。将来、ローカルCVの精度が不十分と判明した場合に限り、ユーザー承認のうえで再検討する。

### 6.2 パイプライン（概略）
1. グレースケール化・二値化（Otsu 等）。
2. 水平投影による**五線検出**（各段の5本線・段位置・線間隔）。
3. 段 → 小節線 → 拍の推定（可能な範囲）。
4. **符頭検出**（テンプレート/連結成分等）と、五線位置への写像。
5. 音部記号・調号の把握（第4章）。
6. 音名エンジンへ入力してプリフィル。

### 6.3 CVの位置づけ
- CV は「プリフィルする補助」であり出力の正しさを保証しない。フォーマットが一定である利点を活かして特化チューニングする。
- CV が弱い/失敗しても、手動アノテーション（第7章）だけでアプリが実用になること。

## 7. UI / UX 仕様

### 7.1 画面構成
- 中央：画像（スクショ）を表示する Canvas。ズーム / パン可能。
- 上に音名・コードのラベルを**重ねて**表示。
- コントロール：画像読込、ラベル密度切替え、音部記号/調号の指定・訂正、修正モード。

### 7.2 ラベル表示
- 音名ラベル：各符頭の近傍に `C4` 等を重ねる。
- コードラベル：小節/拍の上にコード名（候補がある場合は候補も分かる形で）。
- **密度切替え（初期状態は切替え制御）**：`全音表示` / `コードのみ` / `選択した音のみ` をボタンで切替え、ライブの場面に応じて見せ方を変える。

### 7.3 確認・修正（必須）
- 認識結果は**いつでもクリックで訂正**できる。
  - 符頭の追加 / 削除 / 位置調整。
  - 音名の直接修正。
  - 音部記号・調号の指定 / 変更。
  - コードの手動指定 / 候補からの選択。
- 目的が指導であり、誤った音名を見せることが致命的なため、修正のしやすさを最優先する。

## 8. 技術仕様

- **単一 HTML をローカルで開く**構成。ビルド不要・インストール不要・オフライン動作。
- 第一候補は**バニラ JS + Canvas**。
- ライブラリを使う場合は**ローカルに同梱**し、CDN 実行時取得に依存しない（オフライン維持）。
- 音楽理論ロジック（音名・コード）は UI から分離した純関数として実装し、単体テスト可能にする。

---

## 9. 非機能要件

- **プライバシー**：画像・データを外部送信しない。
- **コスト**：無料。ユーザーの事前承認なしに課金の発生する仕組みを入れない。
- **性能**：スクショ一枚の解析が、ライブ指導で待てる時間（目安 数秒以内）で完了すること。
- **可搬性**：主要デスクトップブラウザ（Chrome 等）で `index.html` を開けば動く。

---

## 10. 既知の難所とトレードオフ

- **純ブラウザJSのOMRは限界がある**：ビーム連桁・和音・臨時記号・加線・調号の自動把握は誤りやすい。→ 修正UI必須で担保。フォーマット一定を活かして特化。
- **アルペジオのコード推定は本質的に曖昧**：拍/小節グルーピングと候補提示で対応するが、正解が一意でない区間は残る。
- **調号を誤ると異名同音表記がずれる**：音部記号・調号は手動指定でも確定できる導線を用意する。
- **ラベルの混雑**：大譜表・全音表示は密になる。密度切替えで緩和。
